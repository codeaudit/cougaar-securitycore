Servlet Access Control

·          DamlUriMap and Ontology-EntityInstances.daml

These two configuration files are used to control the relationship between the names of servlets that are understood by the policy and  the names of servlets that are used by the policy engine.  A new policy name for a servlet is introduced into the policy engine by adding a line of the following form to the file Ontology-EntityInstances.daml:

<ultralogEntity:Servlet rdf:ID="CAReadServlet"/>

The DamlUriMap file maps Ultralog names for servlets to the names used by the policy management system.  Each line in this file is either a comment (if it starts with #) or a pair of strings separated by a space.  A typical entry in this file looks like

/$*/CA/Index CAReadServlet

·          DamlUserRoleMap and  Ontology-GroupInstances.daml

These three files are used to managing the relationship between the names of roles as understood by Ultra*Log and the names that the policy engine uses to represent roles.  In order to inform the policy engine about a role, introduce a line of the form

<ultralogGroup:Role rdf:ID="AdministratorRole"/>

in Ontology-GroupInstances.daml.  The DamlUserRoleMap maps names of user roles known to Ultra*Log, e.g.

RearUserDomainComm\AdministratorRole

·          UserFile.xml

This file defines the users for the system, their passwords and their roles.

·          Communities.xml

In order to do access mediation on a servlet, a node must be able to contact the user manager.  A node finds its user manager through the community service.  The node looks for User communities that it belongs to:

<Community Name="RearUserDomainComm" >

    <Attribute ID="CommunityType" Value="User" />

which have a declared user manager in them:

<Entity Name="RearUserAdminAgent" >

      <Attribute ID="EntityType" Value="Agent" />

      <Attribute ID="Role" Value="Member" />

      <Attribute ID="Role" Value="UserManager" />

</Entity>
