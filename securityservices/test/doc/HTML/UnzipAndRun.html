Unzip And Run Configuration Guide

 

This document explains how to install the security services overlay. It assumes the society will be run using the ACME environment.

 

Installation

  1. Download from http://www.cougaar.org or http://www.alpine.bbn.com/~mthome to get the most recent release. Select a version that will be compatible with the security services (e.g. security services B10_2 is compatible with Cougaar B10_2).
  2. Install Cougaar as described in the Cougaar installation guide.
  3. Download and install the ACME scripting environment. https://cvs.ultralog.net/dist/isat_acme_stable.zip. The ZIP file contains an installation guide. Also, more installation information can be found on a Wiki page at https://cvs.ultralog.net/ultralog/AcmeServiceFaq.
  4. Download the security services overlay from https://cvs.ultralog.net/dist/. This page requires a user name and password.
  5. The following zip files are required:
    1. CSI security services B10_2 hourly (Cougaar branch B10_2). The file name should be something like AS_csi_secserv_B10_2.zip. Select the appropriate version of the CSI security_services module. The HEAD is not guaranteed to work properly.
    2. University of Memphis Security Console hourly (Cougaar branch B10_2). The file name should be something like AS_umem.zip.

The CSI “overlay” is not required. This module is used as a temporary repository for code contributed to Cougaar core.

The IHMC Safe overlay is not required. This module is already included in the CSI security services module.

  1. Unzip AS_csi_secserv_B10_2.zip under $COUGAAR_INSTALL_PATH.
  2. Unzip AS_umem.zip under $COUGAAR_INSTALL_PATH.
  3. You may also download other overlays, such as the robustness overlays. However, it is strongly recommended to start with one overlay and verify that the society runs as expected.
  4. Install a communities.xml file that contains the appropriate robustness and security entries. A good communities.xml can be found by downloading the BBN commons overlay ZIP file from https://cvs.ultralog.net/dist/bbn_commons_overlay.zip and extracting configs/common/communties.xml.
  5. Build an XML file with the host-node-agent (HNA) names mappings. The security rules do not depend on host name, but they do depend on node and agent names. The ROOT-CA-NODE must run on its own host; no other node must be running with ROOT-CA-NODE. In addition to the standard logistics nodes, make sure you HNA file has the following node entries:
    1. ROOT-CA-NODE
    2. FWD-CA-NODE
    3. FWD-MGMT-NODE
    4. REAR-CA-NODE
    5. REAR-MGMT-NODE
    6. CONUS-CA-NODE
    7. CONUS-MGMT-NODE
    8. TRANS-CA-NODE
    9. TRANS-MGMT-NODE
  6. Execute the security services rules. When you unzip the security services overlay, the rules are installed under $COUGAAR_INSTALL_PATH/csmart/config/rules/security. Make sure you do not have any error while executing the rules. Some of the rules may require some editing, depending on your environment:
    1. The A26-msg_protection_aspect.rule rule loads the Message Protection Aspect as a component. The order in which aspects are initialized is important: the message protection aspect must be the last aspect in the chain of aspects. If you install more than one aspect (e.g. StatisticsAspect, robustness aspects), you should not use the Message Protection Aspect rule. Instead, you should have one rule that initializes all the aspects. This can be done by adding a rule that overrides the org.cougaar.message.transport.aspects Java Virtual Machine property. For example, add the following rule:

society.each_node do |node|

aspects = "org.cougaar.core.mts.NameSupportTimeoutAspect," +

          "org.cougaar.core.mts.RMISocketControlAspect," +

          "org.cougaar.core.mts.StatisticsAspect," +

          "org.cougaar.core.mts.RMISendTimeoutAspect," +

          "org.cougaar.core.mts.ShowTrafficAspect," +

          "org.cougaar.core.mts.MessageOrderingAspect," +

          "org.cougaar.core.mts.MessageSendHistoryAspect," +

          "org.cougaar.core.mts.acking.MessageAckingAspect," +

          "org.cougaar.core.mts.RTTAspect," +

          "org.cougaar.core.mts.MessageNumberingAspect" +

"org.cougaar.core.security.crytpo.MessageProtectionAspect"

node.override_parameter("-Dorg.cougaar.message.transport.aspects", aspects)

end

    1. Depending on your base HNA file, you may or may not have to edit some JVM properties.  In particular, make sure the following properties are defined:

                                                               i.      -Dorg.cougaar.install.path

                                                             ii.      -Dcougaar.core.persistence.clear

                                                            iii.      -Dorg.cougaar.core.persistence.enable

                                                            iv.      -Xbootclasspath/p

                                                             v.      –Xms

                                                            vi.      –Xmx

                                                          vii.      -XX:ThreadStackSize

    1. Appropriate VM property values depend on your environment. The security_param_tic.rule file contains examples of such VM properties. Those properties can be commented out directly in that file, or a new rule file can be created if necessary.

  1. Once you have generated the society definition file (either as a Ruby or XML file), the society is ready to be executed. You may have to create a startup script, if necessary. Make the following changes to your startup script:
    1. Delete the $COUGAAR_INSTALL_PATH/P directory before every run (unless you want to test rehydration)
    2. Delete the $COUGAAR_INSTALL_PATH/workspace/security directory before every run (unless you want to test rehydration)
    3. Delete files under $COUGAAR_INSTALL_PATH/workspace/log4jlogs before every run
    4. Delete files under $COUGAAR_INSTALL_PATH/log/bootstrap before every run
  2. Execute the society by launching you startup Ruby script. The root CA should be started, then the subordinate CAs, then the nodes.

 

Troubleshooting

If the society does not seem to start properly, check the following potential issues:

  • You should not have any exception in the log4j log files. Do a “grep Exception *.log” under $COUGAAR_INSTALL_PATH/workspace/log4jlogs. The exception might give you a clue about the cause of the problem.
  • Make sure the root CA starts correctly. Launch a browser, then go to http://root-ca-hostname:8800, where root-ca-hostname is the host name where the Root CA is running. You should see a “RootCA” entry in the list of agents. Click on “RootCA”, then select the “Index” servlet. You should see the root CA management interface. Click on the “Certificate List” URL, then select the default CA in the drop-down list. You should see the Root CA key, and four subordinate CA keys, if the subordinate CAs have started correctly.
  • Make sure the subordinate CAs start correctly. After the root CA is initialized, you should see the subordinate CAs if you go to http://root-ca-hostname:8800.
  • Make sure ALL nodes use the same naming service (org.cougaar.name.server property).
