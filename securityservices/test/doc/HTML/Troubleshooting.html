Troubleshooting

 

The society does not start or does not go to completion even without the security services

The first thing to do is to configure and run the society without the security services. Install ACME and Cougaar properly (refer to the ACME and Cougaar documentation). Once you have been able to go to completion without the security services, install the security overlay.

 

The RunSoc.rb script exits with an error message

  • Check that ACME is running on every machine. There should be one instance of Ruby running on every machine.
    1. cd $CIP/operator
    2. ./runall “ps –ef | grep ruby”
  • If ACME daemons are not running properly, type:
    1. cd $CIP/operator
    2. ./reset
    3. ./runall “service acme restart”
  • On some occasion, Java processes are still alive and preventing proper communication between the Jabber server and the ACME daemons. If this is the case, type:
    • cd $CIP/operator
    • ./reset

 

The RunSoc.rb scripts is executed, but the security services do not seem to be enabled

  • First, you must install the security services overlay.
  • You must edit the RunSoc.rb script and add the security rules. Add the following line after the “isat” and “logistics” rules. Do not forget to type the comma as a field separator.
    • #{RULE}/security

 

The nodes die very quickly after startup

  • Configuration files are stored in signed jar files. The Secure Config Finder does not load unsigned files, so the node may not be able to load configuration files. See below, “I have changed a configuration file, but the nodes still use the old data”, on how to package and sign jar files containing configuration data.

 

The log4j log files are not generated

  • Make sure you have a loggingConfig.conf file under $CIP/configs/common
  • Make sure the following properties are set properly:
    • -Dorg.cougaar.core.logging.config.filename
      • value: loggingConfig.conf
    • -Dorg.cougaar.core.logging.log4j.appender.SECURITY.File
      • value: $COUGAAR_INSTALL_PATH/workspace/log4jlogs/#{node.name}.log
  • The log4j VM properties may not be set properly (see comments above). This may cause the Java security manager to block access to log4j log files. For example, you may get access violations in the $CIP/log/bootstrap files:
    • SecurityManager_ROOT-CA-NODE_2003-6-3-4-40.log
      <logtime>      Jul 3, 2003</logtime>
      <nodeName>ROOT-CA-NODE</nodeName>
      <securityEvent><securityManagerAlarm><time>      Jul 3,
      2003      </time><perm>(java.io.FilePermission node.log write)</perm>
      <stack>
      at org.cougaar.core.security.securebootstrap.CougaarSecurityManager
    • The node tried to open the node.log file, but it does not have the permission to open that file. Usually, the node tries to open "node.log" when log4j is not configured properly. For example, you do not have a
      loggingConfig.conf file under $CIP/config/common. Another possibility is that the log4j parameter in the Ruby rules is not set properly. The name of the log4j file is usually set to the name of the node. Open the society XML file and check that the log4j file parameter is set properly. Not being able to log anything means future problems might be hidden.
    • There could be something wrong with the log4j configuration. There should not be a "node.log" file permission. When the log4j file name has not been specified, the name "node.log" is used by default, but the default should always be overridden. Open the society XML file and search for the VM property "org.cougaar.core.logging.log4j.appender" for a node (for example ROOT-CA-NODE). It should be set to something like ROOT-CA-NODE.log.

 

I have changed a configuration file, but the nodes still use the old data

  • Configuration files are stored in signed jar files. The Secure Config Finder does not load unsigned files, so the node may still use an older configuration file contained in a signed jar file. You must package all configuration files in jar files and sign those jar files. Depending on which version of the ACME service you are using, this may or may not be done automatically. One way to quickly sign all the necessary Cougaar configuration files is to use a script provided in the security services:
    • Checkout the security services module
    • cd securityservices
    • ./test/bin/createJarConfigFiles
    • This will package all Cougaar configuration jar files except for the security services config jar file. If you plan to modify a configuration file from the security services, you must recompile and install the security services:
    • ./test/bin/installsecurity -i

 

The RunSoc.rb scripts is executed properly, but the society is never ready for Oplan

  • Delete the $COUGAAR_INSTALL_PATH/workspace directory before every run. This should be done unless you plan to test rehydration. But rehydration should be tested only after you have successfully executed the society without rehydration.
  • First, make sure the root CA starts correctly. Launch a browser, then go to http://root-ca-hostname:8800, where root-ca-hostname is the host name where the Root CA is running. You should see a “RootCA” entry in the list of agents. Click on “RootCA”, then select the “Index” servlet. You should see the root CA management interface. Click on the “Certificate List” URL, then select the default CA in the drop-down list. You should see the Root CA key, and four subordinate CA keys, if the subordinate CAs have started correctly.
  • Make sure the subordinate CAs start correctly. After the root CA is initialized, you should see the subordinate CAs in the list of agents if you go to http://root-ca-hostname:8800. With default rules, you should have four subordinate CAs: Fwd, Rear, Conus and Trans EnclaveCaManager.
  • Make sure all agents are started correctly. If agents are started and get their keys properly, agent should register in the white pages. You should see the list of all agents in the root servlet.
    1. Full-1AD society: there should be about 230 entries when displaying the root servlet.
  • Once the society has started, go to the NCA task servlet, and then display all UniqueObjects. After a few minutes, all NCA subordinates should report for duty. There should be one Asset entry per subordinate. After a while, NCA should receive OplanReadyMsg from each subordinate.
    1. Full-1AD society: NCA has 7 subordinates
      1. USECOM
      2. FORSCOM
      3. HNS
      4. DLAHQ
      5. OSC
      6. TRANSCOM
      7. NATO
  • You should not have any exception in the log4j log files. Do a “grep Exception *.log” under $COUGAAR_INSTALL_PATH/workspace/log4jlogs. The exception might give you a clue about the cause of the problem. A more fine-grained search for problems is done by typing:

grep –E “WARN|ERROR|FATAL|Exception” *.log | grep –v –E “ComplainingLP|STDOUT”

  • Make sure ALL nodes use the same naming service (org.cougaar.name.server property).
  • Some jar files may not be signed properly.
  • The Java security manager may block access to VM resources. In a normal run, no access permission violation should be generated. The average size of the files contained in the $CIP/log/bootstrap should be around 60 bytes. Those files should have the <logtime> and <nodeName> tags only.
    1. cd $CIP/log/bootstrap
    2. ls –l

 

None of the agents in one particular enclave start

  • This is most likely due to a configuration issue with the corresponding enclave CA. If an enclave CA is not
    started, nodes that get their keys from that enclave CA will never be able to participate in the society. There may be a problem with the host where the enclave CA is supposed to run. Check that this machine is up and running, that and that the ACME service is running. There may also be a society configuration issue (e.g., does this enclave CA exist in the society, and does the host specified in the configuration exist?)
  • Check the ACME log files under the ACME installation (usually /usr/local/acme). In particular, check for any problem in the run.log file.

 

The node on one particular machine never starts

  • Is this machine up and running? Can you ping it?
  • Is the ACME service running on that machine?
  • Check the ACME log files under the ACME installation (usually /usr/local/acme). In particular, check for any problem in the run.log file.
  • If you are running NIS, is NIS running on that machine?
  • Can you login as the standard Cougaar user on that machine?
  • Can you login as the standard ACME user on that machine?

 

It looks like certificates are not generated

  • You can check that certificates have been generated by typing the following commands:
    • cd $CIP/workspace/security/keystore
    • ls –l
    • There should be one subdirectory per node. Each node should have a keystore file containing one key for the host, one key for the node, and one key per agent
    • cd $node_name
    • keytool –list –v –kestore keystore-${node_name}

Agents do not register with the nameserver,

  • If agents do not register with the nameserver, no communication will occur because certificates cannot be retrieved. This problem is caused by a problem in the /etc/hosts file. Make sure you do not have the loopback address assoicated with the hostname, If you have an entry in your /etch/hosts file that looks like:
    • 127.0.0.1 <hostname>   localhost.localdomain         localhost
  • change it to:
    • 127.0.0.1  localhost.localdomain    localhost
